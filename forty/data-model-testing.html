Is 'Fit for Purpose'

Using microservice and GraphQL

A recent (circa 2018) contract has me working on data driven microservice architecure; this is where the data is king and the data determines how the enterprise must respond. While evolving the model, to demonstrate the model was satisfying the requirements, I was creating examples manually and validating against JSON Schema (using avj) and then transforming those examples to another formats (using JSONata) and revalidating.

It was quicker than creating the JSON from hand and manually mapping data, but as the number of examples grew and the different aspects of the model grew, so did the number of examples and the need to create even more examples more readily. Having creating more contrived examples, as the data model changed, not only did the new examples need validating, so did all the previus examples to ensure nothing had been broken. And whilst JSON Schema is good, it does not lend itself to rich context semantic checking of data, especially if the data is linked across sources.

This project uses:

  • node.js & nest - to create a rich set of microservices; significantly, the 'local' data source will be wrapped as microservices using nest's strong MVC framework and pipelines to enforce JSON schema validation and automatic transformation/translation to other schema (knwon pipelines) - still using avj and JSONnata.
  • graphql - a query language for APIs (microservices - via nest above), created by Facebook - to fetch data across multiple sources
